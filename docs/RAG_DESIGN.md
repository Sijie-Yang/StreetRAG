# SyntaxRAG 设计思路

## 从 SpatialRAG 借鉴的核心思想

### 1. 混合检索机制（Hybrid Retrieval）

**SpatialRAG 做法：**
- 稀疏检索：基于 SQL 的空间查询（如 "在半径 X 米内"）
- 密集检索：基于向量嵌入的语义匹配

**SyntaxRAG 借鉴：**
- 结构化检索：基于整合度值、长度等数值特征的查询
- 文本检索：基于空间句法描述的语义搜索

### 2. 空间索引优化

**SpatialRAG 做法：**
- 使用空间索引（如 R-tree）加速地理查询
- 向量嵌入地理特征

**SyntaxRAG 借鉴：**
- 按整合度值建立索引（如 integration_R400, R800, R1200）
- 可以按长度、节点度等特征建立多维度索引

### 3. 多层次数据表示

**SpatialRAG 做法：**
- 点级别：单个位置
- 区域级别：聚合特征
- 关系级别：空间关系

**SyntaxRAG 借鉴：**
- Edge 级别：单条街道的特征
- Segment 级别：街道段聚合（按整合度范围）
- Network 级别：整体网络特征

### 4. 查询增强

**SpatialRAG 做法：**
- 将自然语言转换为空间查询
- 例如："附近的餐厅" → SQL: `ST_DWithin(point, POINT(...), 500)`

**SyntaxRAG 借鉴：**
- "高整合度街道" → `integration_R800 > threshold`
- "连接性好的区域" → 基于节点度查询

## 当前实现 vs 改进方向

### 当前简单实现
- ✅ 基础文本索引
- ✅ 关键词匹配
- ❌ 没有向量嵌入
- ❌ 没有数值范围查询
- ❌ 没有空间关系查询

### 可以添加的功能（保持简单）

1. **数值范围查询**
   ```python
   # 查询高整合度街道
   "integration_R800 > 1e-6"
   ```

2. **多条件组合**
   ```python
   # 查询长且高整合度的街道
   "length > 500 AND integration_R800 > threshold"
   ```

3. **向量嵌入（可选）**
   ```python
   # 将空间句法特征转为向量
   embedding = [length, integration_R400, integration_R800, ...]
   ```

4. **聚合查询**
   ```python
   # "哪些区域整合度最高"
   # 按网格或区域聚合整合度值
   ```

## 保持简单的原则

1. **渐进式扩展**：先有基础功能，再逐步添加
2. **避免过度设计**：不需要一开始就用向量数据库
3. **实用优先**：根据实际查询需求添加功能

## 下一步建议

1. 添加数值范围查询支持
2. 添加简单的过滤条件（AND, OR）
3. 可选：添加向量嵌入（使用 sentence-transformers）
4. 可选：添加 LLM 生成回答（基于检索结果）
